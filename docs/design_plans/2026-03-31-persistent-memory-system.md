# Persistent Memory System — Design Discussion Summary

> Date: 2026-03-31
> Status: Draft — for discussion

---

## Goal

Give the agent persistent memory across chat sessions. Two high-level categories:

- **User memories** — facts about the user (profile, preferences, relationships, life events)
- **Agent experience** — what the agent has learned / figured out about the user over time

Both live alongside the productivity subsystem (tasks, projects, journals) but are separate.

---

## Storage

- **Location:** Separate SQLite inside `myproject_core/memory/`, following the same per-user isolation pattern as the productivity subsystem.
- **Schema:** Document model — each memory entry has subject/title + metadata + content, no tension between structured fields and free text.
- **Tables:** Two separate tables, one per memory type.

---

## Two Memory Types

### EventLog (append-only)

Records discrete facts about moments in time. Never overwritten — new events are added, old ones preserved.

**Fields:**

| Field | Type | Notes |
|-------|------|-------|
| `id` | int | Primary key |
| `subject` | string | Optional. For human browsing. |
| `event_time` | datetime | When the event actually happened (may differ from `created_at`) |
| `content` | text | What happened |
| `tags` | list[string] | For retrieval. Agent-generated. User-editable. |
| `importance` | int (1-5) | Agent-assigned and user-overridable |
| `source` | enum | `"agent_tool"` / `"dream_workflow"` / `"user_manual"` |
| `related_memory_ids` | list[int] | Optional links to related topical memories or event logs |
| `created_at` | datetime | |
| `updated_at` | datetime | |

### TopicalMemory (revisable)

Knowledge that accumulates and can be updated as new information arrives. Old versions are marked as superseded (not deleted).

**Fields:**

| Field | Type | Notes |
|-------|------|-------|
| `id` | int | Primary key |
| `subject` | string | Optional. Auto-generated by agent on creation, user-editable. Display label for browsing. |
| `content` | text | Current state of knowledge |
| `tags` | list[string] | For retrieval. Agent-generated. User-editable. |
| `importance` | int (1-5) | Agent-assigned and user-overridable |
| `source` | enum | `"agent_tool"` / `"dream_workflow"` / `"user_manual"` |
| `superseded_by_id` | int | FK to newer TopicalMemory, if this was revised |
| `supersedes_ids` | list[int] | Memory IDs this entry supersedes |
| `created_at` | datetime | |
| `updated_at` | datetime | |

**Revision chain:** The `superseded_by_id` / `supersedes_ids` fields form a chain. Walking the chain backwards gives revision history without storing every version.

**Example:** "favourite dim sum" (id=1, content="dim sum") → new entry (id=2, content="not dim sum anymore", supersedes_ids=[1], superseded_by_id=null). Entry 1 gets `superseded_by_id=2`.

## Key Design Decisions

### Separate tables for EventLog and TopicalMemory

Chosen because the two types have fundamentally different behaviours (append-only vs revisable). Keeps queries clean and avoids nullable chaos.

### Subject is optional, auto-generated, user-editable

Tags handle deterministic retrieval. Subject is a display label for browsing. The agent generates it for readability, the user can fix it.

### Importance: both agent-assigned and user-overridable

Agent assigns based on content analysis. User can manually override.

### Provenance tracking

`source` field tracks how each memory was created: `agent_tool` / `dream_workflow` / `user_manual`. Useful for trust and debugging.

### Event logs do not contaminate user journal

Agent-maintained event log is separate from the user's own journal. Different purpose, different audience.

## Creation

### Explicit agent tool

The agent explicitly calls a tool like `remember_this(content="...", type="event"|"topic", subject="...", tags=[...])`. During the conversation, agent decides what's worth remembering and calls the tool.

### Dream workflow (nightly)

Scheduled job runs at end of day. Consumes the day's chat history, uses an agent to:
- Identify new event logs worth creating
- Identify topical memories that need updating or creating
- Produce new/revised entries

Details (where chat history is stored, how the workflow is invoked) deferred to later.

## Retrieval and Hint Mechanism

### Harness proactively offers hints (minimal)

Before each turn, the harness does lightweight keyword matching:
1. Extract a few nouns/topics from the current conversation
2. Query memory DB — `WHERE subject LIKE '%keyword%' OR tags @> '[keyword]'`
3. Inject minimal hints into the clipboard: `"You have a memory about [subject] (tag match). Say 'recall memory #3' to load it."`

This is deterministic, cheap, no vector embeddings. Failure modes are predictable.

### Agent can also explicitly search

Tools like `search_memories(query)`, `get_memory(id)`, `list_memories(category="event"|"topic")`. Full agent control on top of the harness hints.

### Clipboard integration with TTL/decay

- Memories not actively referenced decay via TTL (same as existing clipboard mechanism)
- When pinned/recalled: summary first, full detail on explicit read
- As TTL reduces: detail decays back to summary before disappearing

## User-Facing Features

- **Browse** — list memories, filter by type/tag/date/importance
- **Edit** — modify subject, content, tags, importance
- **Delete** — remove individual memories
- **Memory wipe** — nuclear option to clear all agent memories about the user (future)
- **Revision history** — walk superseded chain for topical memories (no explicit rollback UI needed — user can see history and edit current state)

## Open Questions (Deferred)

1. **Dream workflow details** — where is chat history stored between sessions? How is the nightly job triggered?
2. **Chat history persistence** — currently `AgentMemory.messages` is in-memory. Does it survive across sessions, or does it get saved somewhere for the dream workflow to consume?
3. **Retrieval strategy experimentation** — the keyword+tag approach is the baseline v1. The pluggable "relevance strategy" layer should allow swapping in different approaches later (e.g., LLM-assisted relevance scoring).
4. **Importance scoring algorithm** — how does the agent automatically assign importance 1-5? Heuristic based on content length, tags, explicit signals from conversation?
5. **Related memory linking** — how does the agent decide to link `related_memory_ids`? Same topic? Same person? Same event chain?

## Relationship to Existing Systems

### AgentClipboard

The existing clipboard handles ephemeral state (files being edited, tool results, pinned productivity entities). Memory is persistent — it survives sessions. Both feed into the same clipboard rendering for the agent's context window.

### Productivity subsystem (Tasks, Projects, Journals)

Memory reads from and writes to the productivity subsystem — journals could inform memory ("user wrote about X"), but the agent's memory about the user is separate from the user's own journal entries.

### AgentMemory

Current `AgentMemory` class manages `messages` (chat history) + `AgentClipboard`. The memory subsystem would be a separate layer the harness coordinates with, not something merged into the existing class.
